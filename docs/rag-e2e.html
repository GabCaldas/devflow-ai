Como o RAG ajuda a IA a “olhar ao redor”

Uma viagem interativa por um Pull Request fictício, do diff recebido até o comentário final.

🕵️‍♀️
Pense no RAG como um detetive com acesso ao arquivo da empresa.O diff é a pista nova. Antes de responder, ele procura no repositório documentos que expliquem essa pista.
Etapa 1 · Entrada

Um Pull Request muda só três linhas

O GitHub entrega o diff. Ele mostra o que mudou, mas não o contrato de get_user().

Diff recebido
diff --git a/app/login.py b/app/login.py
+ async def login(email):
+     user = await get_user(email)
-     return user is not None
+     return user.active
O diff não responde
  • get_user() pode devolver None?
  • Todo usuário possui active?
  • Qual padrão o projeto usa?
Sem RAG: o agente adivinha. Com RAG: pesquisa as respostas no próprio código.
Etapa 2 · Preparação

O repositório vira uma coleção pesquisável

O sistema percorre CODEBASE_PATH, ignora dependências e divide arquivos em blocos de até 80 linhas, com sobreposição de 15.

📁 Repositório

app/login.py
app/users/repository.py ← relevante
app/users/models.py ← relevante
app/payments/service.py
node_modules/ ← ignorado
.git/ ← ignorado

✂️ Chunks gerados

repository.py:1–80repository.py:66–131models.py:1–72service.py:1–80

Cada bloco guarda caminho, linhas e conteúdo. Nada é enviado para uma API de embeddings.

Etapa 3 · Recuperação

Cada chunk recebe uma nota

O diff vira a consulta. Feature hashing cria vetores locais; BM25 valoriza termos raros como get_user e active.

🏁 Ranking

repository.py:1–800,91
models.py:1–720,84
login.py:1–450,76
payments/service.py0,09

Por que esses venceram?

  • repository.py define get_user().
  • models.py define User.active.
  • O arquivo alterado recebe bônus.
  • Payments não traz contexto útil.
nota = 45% vetor + 55% BM25 + bônus
Etapa 4 · Augmentation

O sistema monta o pacote de evidências

Os melhores chunks são anexados ao diff. O limite de 7.000 caracteres protege o free tier.

1 · AlteraçãoDiff: return user.active
2 · Contexto recuperadoget_user(...) -> User | None
class User: active: bool
3 · InstruçãoEncontre bugs e proponha correções usando o diff e o contexto como evidência.
O código recuperado é marcado como referência, não instrução, reduzindo risco de prompt injection.
Etapa 5 · Generation

Os agentes usam a mesma evidência

O Triager recebe só o diff para economizar tokens. Os agentes escolhidos rodam em paralelo com o contexto.

🧐Reviewer

Vê que user pode ser None antes de .active.

🧪Test Generator

Cria testes para usuário inexistente e inativo.

📝Documenter

Documenta retorno e comportamento do login.

Onde entra o LLM? Apenas aqui. Busca, vetorização e ranking rodam localmente e sem custo de API.
Etapa 6 · Saída

O relatório apresenta problema e evidência

O DevFlow AI agrega os resultados e pode publicar o comentário no Pull Request.
